# 天问-AGI 三路路线图
> 版本: v1.0 | 状态: 提案 | 日期: 2026-05-09
> 用户决策: 三路同走，优先观测自动化

---

## 一、战略框架：三路并行，观测优先

```
路线A ──── AGI研究平台 ────────── 认知架构 + 多智能体协作
    │
路线B ──── 观测自动化 ★ ──────── 望远镜控制 + 调度优化 (最高优先级)
    │
路线C ──── 天文知识引擎 ───────── 文献RAG + 假说生成 + 模式发现
```

**资源配置比例建议:**
- 路线A: 10%（维持架构，不做重投入）
- 路线B: 50%（核心差异化，集中资源）
- 路线C: 40%（快速落地，支撑路线B的假说验证）

---

## 二、路线B：观测自动化（最高优先级）

### 2.1 核心目标
打造国内首个**开放天文台自动控制系统**，支持 ASCOM/INDI 多协议，真实控制望远镜完成"假说→观测→发现"闭环。

### 2.2 当前能力

| 模块 | 状态 | 代码位置 |
|------|------|---------|
| INDI 协议客户端 | 有框架，真实连接待测 | telescope/linker.py |
| Seestar MCP 客户端 | 有框架，真实连接待测 | telescope/seestar_client.py |
| MQTT 望远镜桥接 | 有框架 | telescope/mqtt_bridge.py |
| 望远镜模拟器 | 完整 | telescope/simulator.py |
| 观测调度器 | 成熟（综合评分算法） | observation/scheduler.py |
| 观测执行器 | 模拟模式完成 | observation/executor.py |
| 望远镜增强调度 | 有 | telescope/enhanced_scheduler.py |
| 具身智能层 | 有框架 | observation/embodied.py |

### 2.3 缺失环节

```
真实望远镜连接（INDI/KStars/Ekos）
    ↓
真实图像采集（FITS格式处理）
    ↓
真实Plate Solving（astrometry.net / lokalite）
    ↓
真实图像分析（AstroKit / photutils）
    ↓
真实数据回传闭环
```

### 2.4 阶段规划

**B-1: INDI 望远镜集成（D+1~14）**
- 目标: 连接真实望远镜（优先树莓派 + INDI 环境）
- 任务:
  - [ ] 实现 `telescope/indi_client.py`（完整 INDI 协议支持）
  - [ ] 集成 KStars/Ekos 控制流程
  - [ ] 实现望远镜指向校准（极轴校准、高度角限制）
  - [ ] 实现图像采集触发
- 里程碑: 能通过命令行控制望远镜指向指定坐标

**B-2: Plate Solving 集成（D+15~30）**
- 目标: 从图像确定视野中心坐标
- 任务:
  - [ ] 集成 astrometry.net API（在线求解）
  - [ ] 或集成 lokalite（本机求解）
  - [ ] 实现 FITS 图像元数据读取
- 里程碑: 上传 FITS 图像，5秒内返回视野中心 RA/Dec

**B-3: 图像分析管道（D+31~45）**
- 目标: 自动分析观测图像，检测变星/新天体
- 任务:
  - [ ] 使用 `astronomy/pipeline.py` 三阶段检测
  - [ ] 与已知星表交叉比对
  - [ ] 生成检测报告
- 里程碑: 图像 → 检测报告 全自动

**B-4: 完整闭环联调（D+46~60）**
- 目标: 端到端假说→观测→发现
- 任务:
  - [ ] 假说生成触发观测计划
  - [ ] 观测执行并实时推送状态
  - [ ] 数据分析产生发现报告
  - [ ] 发现写入知识库
- 里程碑: 完整闭环成功率 > 50%

### 2.5 望远镜硬件选择

| 方案 | 成本 | 控制协议 | 推荐度 |
|------|------|---------|--------|
| Seestar S50 | 低（~3000元） | WiFi/MCP | ★★★★★ |
| 树莓派 + INDI | 中（~5000元） | INDI | ★★★★ |
| ASCOM + NINA | 高（自备设备） | ASCOM | ★★★ |
| ZWO ASIair | 中（~2000元） | WiFi | ★★★★ |

**推荐起点**: Seestar S50（低成本、可 MCP 控制、社区活跃）

## 三、路线C：天文知识引擎

### 3.1 核心目标
补全"文献调研"闭环，为路线B提供假说来源和发现验证能力。

### 3.2 当前能力

| 模块 | 状态 | 代码位置 |
|------|------|---------|
| arXiv 文献搜索 | 有框架 | research/literature.py |
| 研究现状分析 | 有框架 | research/hypothesis.py |
| 假说生成 | 有框架 | agents/hypothesis_gen.py |
| 假说验证 | 有框架 | agents/hypothesis_test.py |
| 发现追踪 | 有框架 | research/discovery.py |
| ChromaDB RAG | 计划未落地 | memory/rag.py |

### 3.3 缺失环节

- ChromaDB 向量数据库未集成（v3.7.0 P1）
- 文献调研准确率仅 60-80%
- 假说生成质量依赖 RAG 精度
- 无结构化输出（JSON/YAML）

### 3.4 阶段规划

**C-1: ChromaDB RAG 落地（D+1~14）**
- 目标: 文献调研准确率 80% → 92%
- 任务:
  - [ ] 部署 ChromaDB 服务
  - [ ] 实现 sentence-transformers 嵌入
  - [ ] 集成 arXiv / OpenAlex / Semantic Scholar
  - [ ] 实现混合检索（向量+关键词）
  - [ ] 实现结构化输出
- 里程碑: 输入"系外行星大气光谱分析"，返回 5 篇高相关度论文摘要

**C-2: 假说生成质量提升（D+15~30）**
- 目标: 假说生成质量 75% → 88%
- 任务:
  - [ ] 实现跨领域知识关联发现
  - [ ] 整合 StarWhisper 光谱语言模型能力
  - [ ] 实现假说可检验性评分
- 里程碑: 生成的假说可被天文学家评估为"有一定价值"

**C-3: 模式发现增强（D+31~45）**
- 目标: 从观测数据中发现未知模式
- 任务:
  - [ ] 扩展 `data/pipeline.py` 变星检测
  - [ ] 实现异常检测（Isolation Forest / Autoencoder）
  - [ ] 与已知天文目录自动比对
- 里程碑: 从 TESS 数据中发现 1 个新的变星候选体

## 五、三路协同关系

```
路线C: 知识引擎
  ├─ 文献调研 → 生成路线B的观测目标假说
  └─ 发现追踪 ← 接收路线B的观测结果

路线B: 观测自动化
  ├─ 接收路线C的假说 → 执行观测
  └─ 观测数据 → 送路线C分析

路线A: AGI平台
  └─ 为路线B/C 提供架构支撑
```
